디자이너가 시안을 던지고 → 결과물을 다시 받는 답답함을 줄이려고 시도해봤습니다.

왜 이 방식인가

디자인 시스템을 만들고, 그 시스템으로 사이트를 만드는 과정이 한 번에 흘러갑니다. 처음 시도해본 이번 작업에서 가장 좋았던 3가지를 정리하면.

1.

사이트를 만들며
시스템도 같이 다듬는다

원래는 디자인 시스템을 먼저 완성한 뒤 사이트를 만드는 순서였습니다. 이번엔 사이트를 만들어가면서 시스템에 빠진 부분을 발견하고 그때그때 채워넣는 방식입니다. 시스템과 사이트가 같이 자랍니다.

2.

디자인이 코드로
바로 옮겨진다

Claude 가 Figma 시안을 직접 볼 수 있게 연결했습니다. 색·간격·컴포넌트 정보를 디자이너가 일일이 "이건 #xxx 야" 라고 설명하지 않아도, Claude 가 시안에서 읽어와 코드로 옮깁니다.

3.

"왜 이렇게 했지?"
가 나중에도 보인다

Claude 와 주고받은 모든 대화, 그리고 코드 변경 이력이 그대로 보관됩니다. 시간이 지난 뒤 "이 색은 왜 이렇게 정했지?" 같은 질문이 생겨도, 그때의 결정 과정과 이유를 다시 찾아볼 수 있습니다.

01
Phase 01 · Setup

환경 셋업 — 도구와 도구를 잇는다

Claude Code 가 Figma 시안을 "볼 수" 있게 만드는 게 출발. 한 번만 셋업하면 이후 모든 작업에서 자동 활용.

디자이너가 실제로 할 일은 이게 전부:

  • Figma 데스크탑 앱 열기 (이미 있는 그대로)
  • Figma 의 "Dev Mode" 활성화 — 시안의 색·간격 정보를 외부에서 볼 수 있게 해주는 모드
  • Claude Code 에 Figma 연결 (MCP) 한 번 설정 — 처음에만 셋업, 이후 자동
  • 새 빈 폴더 하나 만들기 — Claude 가 여기서 작업

그 외 폴더/파일 구조, 설정 파일들은 모두 Claude 가 알아서 만들어요. 디자이너가 신경 쓸 필요 없습니다.

왜 중요한가

MCP 연결이 없으면 Claude 는 "Figma 시안 사진을 눈으로 보고 추측하는" 식으로 작업해요. 정확도가 뚝 떨어집니다. MCP 가 있어야 Claude 가 시안의 정확한 값(색 hex, 픽셀 간격 등)을 직접 읽을 수 있어요.

02
Phase 02 · Tokens

Figma 변수 → CSS 토큰으로

디자인 시스템의 "원자" — 색, 타이포, 간격, 효과 — 를 코드 변수로 매핑. 한번 잘 잡으면 이후 모든 결정의 기준이 됨.

어떻게 진행하나 — 한 줄이면 됨

Figma 시안을 Claude 에 보여주고 이렇게 말하면 끝:

실제로 던지는 한 마디

"이 Figma 파일 토대로 디자인 시스템 토큰 만들어줘. 색·타이포·간격·둥글기·움직임 다 포함해서, Light/Dark 둘 다."

Claude 가 알아서:

  • Figma 에서 값들을 다 가져옴
  • 의미 있는 이름으로 정리
  • Light/Dark 모드 동시에 정의
  • 나중에 디자이너가 검증하기 쉬운 형식으로 저장

받은 결과를 Figma 와 한번 비교해서 누락/오류 있으면 짚어서 다시 요청하면 됩니다.

시안이 추가/변경될 때

일회성 작업이 아니에요. "Figma 에 새 색 추가했어, 토큰 업데이트해줘" 라고 하면 Claude 가 반영 + "이번에 ○○ 색 3개 추가, ××는 값 변경" 처럼 변경 내역까지 알려줍니다. 디자이너 입장에선 시안만 관리하면 되고 코드는 따라옵니다.

03
Phase 03 · Components

컴포넌트별 인터랙션 매핑

토큰만 있으면 정적. 컴포넌트는 상태 + 동작 + 인터랙션 까지. Figma 의 컴포넌트별 노드 ID 를 따라가며 하나씩 가져와 구축.

어떻게 진행하나 — 역시 한 마디

실제로 던지는 한 마디

"이 Figma 컴포넌트들 코드로 만들어줘. 크기 옵션과 hover/눌림/비활성 상태까지 다 포함해서, Phase 02 토큰 쓰는 걸로."

Claude 가 알아서:

  • Figma 의 각 컴포넌트(버튼·카드·FAQ 등) 를 하나씩 코드로 변환
  • 크기 옵션(큰/중간/작은)과 상태(기본·hover·눌림·비활성) 모두 매핑
  • 색·간격은 Phase 02 의 토큰만 참조해서 작성 (직접 박지 않음)
  • Figma 스크린샷과 결과물을 나란히 비교하면서 검증

차이가 보이면 짚어서 "이 hover 색이 좀 더 진해야 해" 처럼 자연어로 요청하면 됩니다.

컴포넌트가 추가/변경될 때

새 컴포넌트 추가나 기존 변경이 있으면 "이 컴포넌트 새로 추가했어" 또는 "이 버튼 디자인 바꿨어" 라고 요청. Claude 가 반영하고 "○○ 컴포넌트 추가됨 / hover 상태 변경됨" 변경 내역까지 알려줍니다.

04
Phase 04 · Pages

페이지 고도화 — 토큰과 컴포넌트의 조합

토큰과 컴포넌트가 준비됐으니 이제 실제 페이지를 짓는 단계. Hero · Features · FAQ 같은 페이지 단위로 시안을 조합합니다.

어떻게 진행하나 — 페이지 단위로 한 마디

실제로 던지는 한 마디

"이 Figma 페이지 (예: 메인 화면) 보고 만들어줘. 앞서 만든 컴포넌트랑 토큰 그대로 써서."

Claude 가 알아서:

  • Figma 페이지의 레이아웃·콘텐츠 가져와 페이지 구성
  • 앞서 만든 컴포넌트와 토큰만 사용 (직접 박는 값 없음)
  • Figma 시안 스크린샷과 결과물을 나란히 비교하며 검증
  • 차이 발견 시 원인 자동 분리 (토큰 문제 / 컴포넌트 문제 / 페이지 문제)
시스템 보강이 일어나는 순간

페이지 만들다가 "어, 이 사이즈 버튼은 시스템에 없네?" 같은 상황이 생기면 — Claude 가 거꾸로 시스템에 추가하고 알려줍니다. 그러면 시스템도 그만큼 단단해져요. 이게 바로 "사이트 만들며 시스템도 같이 다듬는다" 의 실체.

시안이 추가/변경될 때

"이 페이지 새로 디자인했어" / "Hero 섹션 톤 바꿨어" 라고 요청하면 Claude 가 페이지 업데이트 + 변경 내역까지 알려줍니다.

05
Phase 05 · System Docs

시스템 자신을 문서화

"이 사이트를 짓기 위해 사용한 시스템" 을 그 사이트 안에서 보여주는 메타 페이지. 디자인 시스템이 살아있는 문서가 됨.

만든 메타 페이지 4종:

  • Foundation — 모든 색 토큰을 light/dark 듀얼 칩으로 나란히. 타이포 스케일 표 (% 단위 line-height). 간격·둥글기·모션 토큰.
  • Layout — GNB · SNB · TOC · Footer · Page Grid 구조도와 변수.
  • Components — 각 컴포넌트마다: 토큰 정의 → 라이브 데모 → 마크업 템플릿 → 실제 사용처 링크 4-part 구조.
  • Audit — 토큰 coverage 표. ✓/⚠/✗ 로 누락 점검.
왜 이게 강력한가

디자인 시스템 사이트의 "system 페이지가 곧 살아있는 spec" 이 됨. 디자이너가 변경하면 Figma 와 system 페이지가 함께 갱신 → 항상 최신 상태 보장.

06
Phase 06 · Sandbox

검증 — 시스템은 충분히 유연한가

진짜 디자인 시스템이라면 같은 콘텐츠로 여러 시안을 빠르게 만들 수 있어야 함. Sandbox 페이지로 검증.

구성:

  • System 그룹 (A/B/C) — 시스템 토큰 100% 내부, 레이아웃과 UX 만 다르게
  • Free 그룹 (D/E/F) — 사용자가 명시한 톤 (Glow Dark, Bento Vibrant, Pixel Kitsch), 시스템 무관
  • 사이드 플로팅 토글로 시안 간 즉시 이동

배운 점

Lesson 01

Figma 연결을 처음에 잘 잡아두세요

  • 연결(MCP) 안 돼 있으면 Claude 는 시안을 눈으로 보고 추측하는 식이 됨
  • 정확도가 뚝 떨어짐
  • 시간 들여서라도 처음에 연결 잘 잡는 게 이후 모든 작업의 효율을 결정
Lesson 02

색·간격 이름은 "의미" 중심으로

  • "회색-900" 같은 외형 이름 말고 "주요-텍스트" 같은 의미 이름이 좋음
  • 나중에 디자인이 바뀌어도 이름이 유지돼서 시스템이 흔들리지 않음
Lesson 03

시스템 문서 페이지가 곧 살아있는 명세서

  • Foundation/Layout/Components 페이지를 만들어두면, 그게 디자이너·개발자 모두 보는 진짜 매뉴얼이 됨
  • 디자인 변경되면 이 페이지도 같이 갱신
  • 항상 최신 상태 유지
Lesson 04

마지막에 "시안과 코드가 진짜 매칭됐는지" 한 번 더 점검

  • 결과물이 멀쩡해 보여도 안에 빈틈이 있을 수 있음
  • 예: Figma 에 정의한 색 중 코드엔 안 들어간 게 있을 수 있음
  • 예: 코드에 색 값을 직접 박아둬서 시스템과 따로 노는 부분이 숨어있을 수 있음
  • 끝에 꼭 한 번 점검 필수
Lesson 05

"트렌디하게" 보다 "Linear 스타일로" 가 훨씬 효과적

  • Claude 는 "안전한 평균값" 으로 가려는 경향이 있음
  • "트렌디하게" 라고 하면 작년 평균값을 내놓음
  • 자유로운 시안일수록 구체적 레퍼런스(이름)를 주면 훨씬 좋은 결과
Lesson 06

잘 만든 작업 흐름은 다음 프로젝트에 그대로 쓸 수 있음

  • 이번에 합의한 6단계 흐름을 Claude 에 기억시켜둠 (Skill)
  • 다음 프로젝트도 "디자인 시스템 사이트 만들자" 한 마디로 같은 품질의 작업이 시작됨
