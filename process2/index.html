Case Study

Claude Code 로
디자인 시스템 사이트 구축하기

시안에서 디자인 시스템을 추출하고, 실제 사이트까지 구현해본 기록.

Project TGC Design System
Duration 1일
Stack Figma · Claude Code · MCP
Background

프로젝트 배경

제품이나 웹사이트를 만들 때 디자인 시스템은 거의 필수다. 색상, 타이포그래피, 간격, 컴포넌트 규칙 등이 정리되어 있어야 디자인과 개발 모두 일관성 있게 작업할 수 있기 때문이다.

그래서 일반적으로는 디자인 시스템을 먼저 만들고 화면을 설계하는 방식이 이상적인 프로세스로 여겨진다. 하지만 실제 실무는 생각보다 그렇게 흘러가지 않는다.

우리는 어떤 컴포넌트가 필요한지, 어떤 패턴이 반복되는지, 어떤 규칙이 필요한지를 결국 화면을 만들면서 발견하는 경우가 많다. 즉,

  • 디자인 시스템이 있어야 화면을 만들기 쉽고
  • 화면을 만들어야 어떤 디자인 시스템이 필요한지 알 수 있다

그래서 대부분의 프로젝트에서는 디자인 시스템 구축과 화면 설계가 병렬로 진행된다. 문제는 여기서 발생한다. 초기에 정의한 디자인 시스템을 개발에 전달하더라도 실제 화면이 구체화되면서 컴포넌트와 규칙이 계속 변경된다. 결국 디자인 시스템도 수정되고, 개발도 수정되고, 디자인도 수정되는 일이 반복된다.

이 과정은 너무 익숙하지만 늘 비효율적이라는 생각이 들었다. 그래서 이런 질문이 생겼다.

굳이 처음부터 디자인 시스템을 정의할 필요가 있을까?

좋은 시안을 먼저 만들고, AI 가 거기서 디자인 시스템을 추출하면 어떨까?

시안이 바뀌면 시스템도 같이 업데이트되면 어떨까?

마침 TGC 디자인 시스템 사이트를 구축하게 되었고, 이번에는 단순히 디자인 시스템 사이트를 만드는 것이 아니라, 실제 시안을 기반으로 Claude Code 가 디자인 시스템을 생성하고, 그 시스템을 발전시키면서 사이트를 구축하는 방식을 테스트해보기로 했다.

결과적으로 이번 프로젝트는 디자인 시스템 사이트 제작 프로젝트이면서 동시에, AI 와 함께 디자인 시스템을 구축하고 운영하는 새로운 작업 방식을 검증하는 케이스 스터디가 되었다.

Goals

목표

이번 작업의 목표는 크게 세 가지였다.

01

Claude Code 로 실제 사이트를 구축할 수 있는가

Figma 시안을 기반으로 디자이너가 직접 사이트를 구축할 수 있는지 확인한다.

02

시안만으로 디자인 시스템을 생성할 수 있는가

토큰을 하나씩 정의하지 않고, 시안에서 디자인 시스템을 추출할 수 있는지 확인한다.

03

생성된 시스템은 실제로 활용 가능한가

사이트 구축과 추가 화면 제작 과정에서 재사용성과 확장성을 검증한다.

Process

전체 프로세스

Figma 시안 제작
Claude + Figma MCP 연결
디자인 시스템 생성
시스템 고도화
페이지 구축
베리에이션 테스트
Steps

단계별 작업

01
Step 01

Figma 와 Claude 연결

가장 먼저 Claude Code 와 Figma 를 MCP 로 연결했다. MCP 가 연결되면 Claude 는 단순히 화면 이미지를 보는 것이 아니라 실제 디자인 데이터를 읽을 수 있다. 예를 들어

  • Color
  • Typography
  • Spacing
  • Auto Layout
  • Component Structure

등을 직접 분석할 수 있다.

실무 인사이트

MCP 연결 여부가 결과물 품질을 결정한다.

MCP 가 없으면 Claude 는 시안을 추측한다. MCP 가 있으면 Claude 는 실제 디자인 데이터를 기반으로 작업한다. 생성되는 코드 품질과 정확도 차이가 매우 크다.

02
Step 02

시안을 기반으로 디자인 시스템 생성

일반적인 방식처럼 토큰 정의 → Foundation → Component → Page 순서로 진행하지 않았다. 먼저 디자인 시스템 사이트 시안을 만들고 Claude 에게 요청했다.

"이 시안 기준으로 디자인 시스템 만들어줘."

Claude 는 시안을 분석해 Color · Typography · Spacing · Radius · Motion · Component 구조를 자동으로 생성했다. 흥미로웠던 점은 단순히 값을 추출하는 수준이 아니었다는 점이다.

예를 들어 Gray-900 / Gray-800 / Gray-700 같은 값 중심 구조 대신, Text Primary / Text Secondary / Surface Default 처럼 의미 기반(Semantic) 구조로 정리했다. 토큰 네이밍, 계층 구조, Light/Dark 구성까지 상당 부분 자동으로 생성되었다.

실무 인사이트

생각보다 토큰을 먼저 고민할 필요가 없었다.

좋은 시안을 먼저 만들고, Claude 가 그 안의 규칙을 추출하도록 하는 방식이 훨씬 빠르게 시작할 수 있었다.

03
Step 03

디자인 시스템 고도화

생성된 시스템이 끝은 아니었다. 실제 페이지를 만들기 시작하면 빠진 부분이 보이기 시작한다. 예를 들어

  • 버튼 상태 부족
  • 카드 패턴 부족
  • 레이아웃 규칙 부족
  • 예외 케이스 누락

등이 발생한다. 페이지를 구현하면서 필요한 부분을 다시 시스템에 반영했다.

실무 인사이트

디자인 시스템은 처음부터 완벽하게 설계하는 것보다, 실제 페이지를 만들면서 발전시키는 방식이 더 현실적이었다.

오히려 실제 사용 과정에서 발견되는 규칙들이 훨씬 실용적이었다.

04
Step 04

컴포넌트 고도화

이번 작업에서 가장 시간이 많이 들어간 단계다. Foundation 생성은 생각보다 잘 되었지만 컴포넌트는 별도 작업이 필요했다.

Claude 가 기본 구조를 생성한 뒤, MCP 를 통해 개별 컴포넌트를 다시 연결하며 세부 상태를 보완했다. 예를 들어

  • Hover
  • Active
  • Disabled
  • Focus

등을 추가로 정의했다. 버튼, 카드, FAQ, 레이아웃 컴포넌트 등을 반복적으로 수정하며 실제 서비스 수준으로 다듬어 나갔다.

실무 인사이트

토큰보다 중요한 것은 컴포넌트 품질이었다.

실제 사용성은 결국 상태와 인터랙션을 얼마나 세밀하게 정의하느냐에 따라 결정된다.

05
Step 05

페이지 구축

정리된 시스템을 기반으로 실제 사이트를 구축했다. 구축한 주요 페이지는 다음과 같다.

  • Main
  • Foundation
  • Components
  • Layout
  • Pattern
  • Audit
  • Sandbox

흥미로운 점은 사이트를 만들면서 시스템도 함께 발전했다는 것이다. 페이지 구현 중 발견된 패턴과 규칙을 다시 시스템에 반영했고, 업데이트된 시스템을 다시 페이지에 적용했다.

결국 단방향이 아닌 순환에 가까웠다:

일반적인 흐름
시스템 구축
사이트 제작
실제 흐름
사이트 제작
시스템 보완
사이트 개선
시스템 보완
실무 인사이트

사이트와 시스템은 별개가 아니었다.

실제로는 서로를 검증하고 발전시키는 관계에 가까웠다.

06
Step 06

베리에이션 테스트

마지막으로 생성된 시스템이 실제로 얼마나 다양한 화면을 만들 수 있는지 테스트했다. 동일한 시스템을 기반으로

  • Modern Minimal
  • Asymmetric Layout
  • Inverse Accent

등 여러 버전의 시안을 제작했다. 또한 시스템과 무관한 스타일도 별도로 생성해 비교했다.

실무 인사이트

같은 시스템을 사용하면 시각적 결과도 어느 정도 비슷해진다.

결국 다양성은 토큰·컴포넌트보다 레이아웃 · UX 패턴 · 콘텐츠 구조에서 만들어진다는 점을 확인할 수 있었다.

Key Insight

가장 중요했던 인사이트

Key Insight

한 번에 만들지 말고
쪼개서 만들어라

처음에는 페이지 전체를 한 번에 구현하려고 했다. 하지만 페이지 규모가 커질수록 컴포넌트 해석 오차, 간격 불일치, 스타일 누락, 구조 왜곡 등이 발생했다.

반대로 아래 순서로 작업했을 때 결과물 품질이 훨씬 안정적이었다.

Foundation
Components
GNB

전체 페이지 검증

Claude 는 큰 문제를 한 번에 해결할 때보다 작은 단위로 나누어 해결할 때 훨씬 정확하다.

특히 디자인 구현에서는 토큰 → 컴포넌트 → 레이아웃 → 페이지 순서로 점진적으로 쌓아가는 방식이 원본 시안과의 일치율이 가장 높았다. 이번 프로젝트에서 가장 효과가 좋았던 작업 방식이기도 하다.

What Worked

좋았던 점

시안이 빠르게 구현된다

디자인 → 개발 전달 과정이 크게 줄어든다.

수정 속도가 빠르다

디자인 수정과 코드 수정 사이클이 매우 짧다.

시스템이 자연스럽게 만들어진다

페이지를 만들다 보면 필요한 시스템이 보인다.

결정 과정이 남는다

Claude 와의 대화, 코드 변경 이력이 모두 기록된다. 나중에 왜 그렇게 결정했는지 다시 확인할 수 있다.

Lessons

배운 점

MCP 연결은 필수다

Claude 가 추측하게 만들지 말고 데이터를 읽게 만들어야 한다.

시안이 생각보다 중요하다

좋은 디자인 시스템보다 좋은 시안이 먼저다. 시안 품질이 시스템 품질을 결정한다.

AI 는 구조화에 강하다

패턴 발견, 토큰 생성, 시스템 정리는 생각보다 높은 수준으로 수행한다.

컴포넌트는 결국 사람이 다듬어야 한다

상태, 인터랙션, 사용성은 여전히 사람의 역할이 중요하다.

사이트를 만들면서 시스템도 성장한다

처음부터 완벽한 시스템을 만드는 것보다 실제 사용 과정에서 발전시키는 방식이 현실적이다.

Closing

정리

이번 작업은 디자인 시스템 사이트를 만든 프로젝트이기도 하지만, 실제로는 Claude Code 를 활용한 새로운 웹 구축 프로세스를 검증한 작업에 더 가까웠다.

특히 인상적이었던 점은 디자인 시스템을 먼저 정의하고 페이지를 만드는 방식보다, 시안을 먼저 만들고 → 시스템을 추출하고 → 시스템을 고도화하면서 → 페이지를 구축하는 흐름이 실제 업무 방식과 훨씬 자연스럽게 맞아떨어졌다는 것이다.

앞으로 Claude Code 를 활용한다면 처음부터 토큰과 컴포넌트를 설계하기보다, 좋은 시안을 먼저 만들고 그 안의 규칙을 시스템으로 발전시키는 접근이 더 현실적인 방법이 될 수 있다고 느꼈다.
