프로젝트 배경

디자인 시스템과 화면은 서로가 서로를 필요로 한다. 시스템이 있어야 화면을 만들기 쉽고, 화면을 만들어야 어떤 시스템이 필요한지 알 수 있기 때문이다.

그래서 대부분의 프로젝트에서는 두 작업이 병렬로 진행된다. 하지만 화면이 구체화될수록 시스템·디자인·개발 모두 반복적으로 수정되는 일이 발생한다. 익숙하지만 늘 비효율적이라고 느꼈다.

그래서 이런 질문이 생겼다.

굳이 처음부터 디자인 시스템을 정의할 필요가 있을까?

좋은 시안을 먼저 만들고, AI 가 거기서 디자인 시스템을 추출하면 어떨까?

시안이 바뀌면 시스템도 같이 업데이트되면 어떨까?

기존 방식

시스템 → 화면
(끝없는 수정 반복)

디자인 시스템 정의
화면 설계
변경 발생 → 화면 수정
시스템도 다시 수정

↺ 디자인·시스템·개발 모두 반복

변화 — 이번 시도

시안 → AI 추출
(시스템·사이트 동반 진화)

좋은 시안 먼저 제작
AI 가 시스템 추출
시안 변경 → 시스템 자동 갱신
시스템·사이트 함께 진화

→ 자연스러운 한 방향 흐름

마침 MIDAS AX System 사이트를 구축하게 되었고, 이번에는 단순히 디자인 시스템 사이트를 만드는 것이 아니라, 실제 시안을 기반으로 Claude Code 가 디자인 시스템을 생성하고, 그 시스템을 발전시키면서 사이트를 구축하는 방식을 테스트해보기로 했다.

결과적으로 이번 프로젝트는 디자인 시스템 사이트 제작 프로젝트이면서 동시에, AI 와 함께 디자인 시스템을 구축하고 운영하는 새로운 작업 방식을 검증하는 케이스 스터디가 되었다.

목표

이번 작업의 목표는 크게 세 가지였다.

01

Claude Code 로 실제 사이트를 구축할 수 있는가

Figma 시안을 기반으로 디자이너가 직접 사이트를 구축할 수 있는지 확인한다.

02

시안만으로 디자인 시스템을 생성할 수 있는가

토큰을 하나씩 정의하지 않고, 시안에서 디자인 시스템을 추출할 수 있는지 확인한다.

03

생성된 시스템은 실제로 활용 가능한가

사이트 구축과 추가 화면 제작 과정에서 재사용성과 확장성을 검증한다.

단계별 작업

01Figma 와 Claude 연결

가장 먼저 Claude Code 와 Figma 를 MCP 로 연결했다. MCP 가 연결되면 Claude 는 단순히 화면 이미지를 보는 것이 아니라 실제 디자인 데이터를 읽을 수 있다. 예를 들어

  • Color
  • Typography
  • Spacing
  • Auto Layout
  • Component Structure

등을 직접 분석할 수 있다.

실무 인사이트

MCP 연결 여부가 결과물 품질을 결정한다.

MCP 가 없으면 Claude 는 시안을 추측한다. MCP 가 있으면 Claude 는 실제 디자인 데이터를 기반으로 작업한다. 생성되는 코드 품질과 정확도 차이가 매우 크다.

02시안을 기반으로 디자인 시스템 생성

일반적인 방식처럼 토큰 정의 → Foundation → Component → Page 순서로 진행하지 않았다. 먼저 디자인 시스템 사이트 시안을 만들고 Claude 에게 요청했다.

"이 시안 기준으로 디자인 시스템 만들어줘."

Claude 는 시안을 분석해 Color · Typography · Spacing · Radius · Motion · Component 구조를 자동으로 생성했다. 흥미로웠던 점은 단순히 값을 추출하는 수준이 아니었다는 점이다.

예를 들어 Gray-900 / Gray-800 / Gray-700 같은 값 중심 구조 대신, Text Primary / Text Secondary / Surface Default 처럼 의미 기반(Semantic) 구조로 정리했다. 토큰 네이밍, 계층 구조, Light/Dark 구성까지 상당 부분 자동으로 생성되었다.

실무 인사이트

생각보다 토큰을 먼저 고민할 필요가 없었다.

좋은 시안을 먼저 만들고, Claude 가 그 안의 규칙을 추출하도록 하는 방식이 훨씬 빠르게 시작할 수 있었다.

03디자인 시스템 고도화

생성된 시스템이 끝은 아니었다. 실제 페이지를 만들기 시작하면 빠진 부분이 보이기 시작한다. 예를 들어

  • 버튼 상태 부족
  • 카드 패턴 부족
  • 레이아웃 규칙 부족
  • 예외 케이스 누락

등이 발생한다. 페이지를 구현하면서 필요한 부분을 다시 시스템에 반영했다.

실무 인사이트

디자인 시스템은 처음부터 완벽하게 설계하는 것보다, 실제 페이지를 만들면서 발전시키는 방식이 더 현실적이었다.

오히려 실제 사용 과정에서 발견되는 규칙들이 훨씬 실용적이었다.

04컴포넌트 고도화

이번 작업에서 가장 시간이 많이 들어간 단계다. Foundation 생성은 생각보다 잘 되었지만 컴포넌트는 별도 작업이 필요했다.

Claude 가 기본 구조를 생성한 뒤, MCP 를 통해 개별 컴포넌트를 다시 연결하며 세부 상태를 보완했다. 예를 들어

  • Hover
  • Active
  • Disabled
  • Focus

등을 추가로 정의했다. 버튼, 카드, FAQ, 레이아웃 컴포넌트 등을 반복적으로 수정하며 실제 서비스 수준으로 다듬어 나갔다.

실무 인사이트

토큰보다 중요한 것은 컴포넌트 품질이었다.

실제 사용성은 결국 상태와 인터랙션을 얼마나 세밀하게 정의하느냐에 따라 결정된다.

05페이지 구축

정리된 시스템을 기반으로 실제 사이트를 구축했다. 구축한 주요 페이지는 다음과 같다.

  • Main
  • Foundation
  • Components
  • Layout
  • Pattern
  • Audit
  • Sandbox

흥미로운 점은 사이트를 만들면서 시스템도 함께 발전했다는 것이다. 페이지 구현 중 발견된 패턴과 규칙을 다시 시스템에 반영했고, 업데이트된 시스템을 다시 페이지에 적용했다.

결국 단방향이 아닌 순환에 가까웠다:

일반적인 흐름
시스템 구축
사이트 제작
실제 흐름
사이트 제작
시스템 보완
사이트 개선
시스템 보완
실무 인사이트

사이트와 시스템은 별개가 아니었다.

실제로는 서로를 검증하고 발전시키는 관계에 가까웠다.

인사이트

이번 프로젝트로 얻은 깨달음을 두 갈래로 정리했다. 작업하며 발견한 작은 인사이트들과, 그것들을 종합해 도달한 결론적 통찰.

A · 작업하며 발견한 것

3가지 작업 인사이트

01

시스템은 다듬어가며 완성된다

시안으로 한 번에 추출할 수 있지만, 실제 페이지를 만들어가다 보면 빠진 규칙과 새로운 패턴이 계속 발견된다. 사이트를 만들면서 시스템도 함께 다듬어진다.

02

화면은 작은 단위부터 쌓아 올린다

페이지 전체를 한 번에 만들면 컴포넌트 해석 오차·간격 불일치·구조 왜곡이 발생한다. 토큰 → 컴포넌트 → 레이아웃 → 페이지 순으로 차근차근.

03

Claude 는 작은 단위에서 더 정확하다

큰 문제를 한 번에 해결할 때보다 작은 단위로 나눠 해결할 때 훨씬 정확했다. 작은 단위부터 쌓아가는 방식이 원본 시안과의 일치율이 가장 높았다.

B · 결국 깨달은 핵심

왜 AI 구축에 디자인 시스템이 필요한가

위 3가지 발견을 종합해보니, 이번 프로젝트의 진짜 결론은 이것이었다. AI 가 단순히 시안 한 장을 보고 코드를 짜는 게 아니라, 디자인 시스템 자체가 AI 에게 "설계도" 역할을 한다는 것. 무엇이 어디에 있고, 어떻게 생겼고, 그 값이 무엇을 의미하는지 — 모두 정리돼 있어야 AI 가 정확히 해석한다.

01 · 계층

AI 가 디자인 맥락을 파악하는 4단계 구조

디자인 시스템은 가장 작은 원자 값(Primitive) 부터 가장 큰 화면 단위(Compositions) 까지 4개 층으로 정리된다. 이 계층이 정확히 자리잡아야 AI 가 어디서 어디로 해석해야 하는지를 안다.

01

Compositions

페이지 = 컴포넌트의 조합

건물 전체 확인
02

Components

구조와 스타일

공간의 역할 (사무실 · 창고 · 화장실)
03

Semantic Tokens

의미 토큰 (Color/Brand · Spacing/md)

자재의 용도 (외장재 · 내장재)
04

Primitive Tokens

원자 값 (#3B82F6 · 16px)

콘크리트 · 벽돌 · 못
02 · 설계도

원자 값만으로는 부족하다

원자재(색·간격·폰트 같은 Primitive Tokens) 만 던져주면 AI 가 무엇을 만들지 모른다. 원자재를 활용할 수 있게 "설계도(디자인 시스템)" 가 함께 전달돼야 한다.

03 · 도면

설계도는 3개 도면으로 구성된다

설계도(디자인 시스템) 안에는 평면도 · 마감 시방서 · 자재 명세서 세 도면이 들어 있다. 셋 다 정리돼 있어야 AI 가 같은 의도로 짓는다. 코드 관점으로 보면 각각 HTML · CSS · JSON 에 대응된다.

1

평면도 HTML / TSX

"뭐가 어디에 있지?"

공간(컴포넌트) 배치와 계층. 건물 평면도처럼 AI 가 페이지를 조립할 때의 첫 단서.

2

마감 시방서 CSS

"어떻게 생겼지?"

외관·재질·치수. 어디에 어떤 마감재를 쓸지 — 색·간격·폰트가 여기에 해당.

3

자재 명세서 JSON

"이 값의 의미가 뭐지?"

자재 자체의 정의·의미. Brand·Spacing/md 같은 토큰이 "어떤 자재인지" 를 알려준다.

→ 평면도 + 마감 시방서 + 자재 명세서 가 한 시스템 안에 정리돼 있을 때, AI 는 설계도 그대로 짓는다.

정리

이번 작업은 디자인 시스템 사이트를 만든 프로젝트이기도 하지만, 실제로는 Claude Code 를 활용한 새로운 웹 구축 프로세스를 검증한 작업에 더 가까웠다.

특히 인상적이었던 점은 디자인 시스템을 먼저 정의하고 페이지를 만드는 방식보다, 시안을 먼저 만들고 → 시스템을 추출하고 → 시스템을 고도화하면서 → 페이지를 구축하는 흐름이 실제 업무 방식과 훨씬 자연스럽게 맞아떨어졌다는 것이다.

앞으로 Claude Code 를 활용한다면 처음부터 토큰과 컴포넌트를 설계하기보다, 좋은 시안을 먼저 만들고 그 안의 규칙을 시스템으로 발전시키는 접근이 더 현실적인 방법이 될 수 있다고 느꼈다.
